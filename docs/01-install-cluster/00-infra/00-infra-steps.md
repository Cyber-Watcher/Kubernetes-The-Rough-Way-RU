# **BIG PICTURE — что мы делаем в этом документе**

Этот документ описывает **Этап 0 — подготовку инфраструктуры** для Kubernetes The Rough Way.  
На этом этапе мы создаём виртуальную среду, которая станет фундаментом всего кластера.

Мы:

- создаём изолированную виртуальную сеть (vmnet) для кластера;  
- вручную создаём три базовые виртуальные машины:  
  - `admin-wks` — рабочая станция администратора,  
  - `cp-01` — шаблон control plane,  
  - `worker-01` — шаблон worker;  
- настраиваем admin‑wks как центральную точку управления;  
- полностью подготавливаем cp‑01 и worker‑01 (ОС, LVM, sysctl, IPVS, journald, containerd, каталоги, пользователи);  
- клонируем cp‑01 → cp‑02/03 и worker‑01 → worker‑02;  
- на каждом клоне меняем MAC, hostname и статический IP;  
- проверяем сеть, ARP, SSH;  
- убеждаемся, что инфраструктура готова к следующему этапу.

**Цель:** получить полностью готовую виртуальную инфраструктуру из 6 машин, на которой можно вручную развернуть Kubernetes.

# **Шаг 0. Подготовка vmnet**

Здесь мы решаем фундаментальную вещь: **в какой L2/L3‑среде вообще будет жить кластер**. Если накосячить на этом уровне, потом Calico, VIP, BGP и даже банальный `ping` будут вести себя странно, и ты будешь ловить фантомные баги.

---

## 0.1. Почему нужна отдельная сеть

**Идея:** кластер должен жить в **предсказуемой, изолированной сети**, где:

- **нет случайного DHCP** от роутера/провайдера;  
- **нет конфликтов IP** с другими устройствами;  
- **мы полностью контролируем CIDR, шлюз, MTU**;  
- **VIP (VRRP) и BGP** не пересекаются с внешней инфраструктурой.

Для нас это важно потому что:

- Calico в режиме BGP/no‑encap будет анонсировать маршруты — если сеть общая, можно легко словить конфликт с реальной сетью;  
- VIP через VRRP (Keepalived) использует **gratuitous ARP** — в общей сети это может конфликтовать с реальными устройствами;  
- IPVS и kube‑proxy завязаны на корректный ARP/route — если вокруг хаос, отладка превращается в ад.

**Вывод:** отдельная vmnet‑сеть даёт нам **лабораторный стенд**, где всё под контролем.

---

## 0.2. Ограничения VMware Workstation на Windows

На **Linux‑версии VMware Workstation** можно создавать несколько кастомных vmnet‑сетей (vmnet2, vmnet3, …, vmnet9 и т.д.), задавать им свои подсети, режимы (Host‑only, NAT, Custom).

На **Windows‑версии VMware Workstation** исторически:

- нельзя было создавать произвольное количество vmnet‑сетей через GUI;  
- обычно доступны vmnet0 (bridged), vmnet1 (host‑only), vmnet8 (NAT);  
- расширенная настройка требует плясок с бубном и не всегда стабильна.

Для нашего проекта:

- **на Linux** мы создаём, например, `vmnet88` с подсетью `192.168.88.0/24`;  
- **на Windows** мы **фиксируем**, что используем `vmnet8 (NAT)` и просто настраиваем статические IP внутри этой подсети.

Мы это явно прописали в этом документе, чтобы человек на Windows не пытался повторить шаги по созданию `vmnet88`, которые он сделать просто не сможет.

---

### 0.3. Создание vmnet на VMware Workstation

Здесь мы описываем **GUI‑шаги** (без CLI‑магии), чтобы любой мог повторить:

1. Открыть **Virtual Network Editor**.  
2. Создать новую сеть, например `vmnet88`.  
3. Выбрать режим **NAT**.  
4. Задать подсеть, например `192.168.88.0/24`.
5. После задания подсети посмотреть Gateway IP. По умолчанию должен быть `192.168.88.2`.
6. Встроенный DHCP можно оставить, так как он пригодится для начальной установки и настройки ОС.  
7. Сохранить настройки.

**Почему так:**

- мы хотим **фиксированный CIDR**, который не меняется от машины к машине;  
- мы хотим, чтобы **все ноды были в одном L2‑сегменте** — это важно для VRRP (VIP) и для BGP (Calico);  
- встроенный DHCP раздает IP из диапазона который не пересекается с нашими IP.

---

### 0.4. Выбор CIDR, VIP, IP‑адресов

Мы фиксируем конкретную схему адресации, чтобы:

- все дальнейшие шаги могли ссылаться на **одни и те же IP**;  
- не было «плавающих» примеров;  
- можно было легко рисовать схемы и дебажить.

Пример (который мы и будем использовать в документации):

- **Подсеть vmnet:** `192.168.88.0/24`  
- **Gateway IP:** `192.168.88.2` 
- **VIP API (VRRP):** `192.168.88.10`  
- **Control plane:**
  - `cp-01`: `192.168.88.11`  
  - `cp-02`: `192.168.88.12`  
  - `cp-03`: `192.168.88.13`  
- **Workers:**
  - `worker-01`: `192.168.88.21`  
  - `worker-02`: `192.168.88.22`  
- **Admin‑wks:** `192.168.88.9`

**Почему такие диапазоны:**

- VIP рядом с control plane, но отдельным адресом;  
- control plane и workers разделены по десяткам (`.1x` и `.2x`) — визуально проще;  
- admin‑wks вынесен в `.9`, чтобы не путать с нодами.

---

### 0.5. Проверка vmnet

Перед тем как вообще трогать ВМ, важно убедиться, что:

- vmnet‑интерфейс **существует** на хосте (например, `vmnet88`);  
- у него есть **правильный IP и маска**;  
- он не конфликтует с другими сетями на хосте.

На этом шаге мы:

- проверяем список интерфейсов;  
- смотрим IP‑адрес vmnet;  
- убеждаемся, что подсеть `192.168.88.0/24` нигде больше не используется.

**Почему это важно:**

- если подсеть пересекается с реальной сетью (например, у тебя дома роутер тоже раздаёт `192.168.88.0/24`), начнутся странные маршруты;  
- если vmnet не поднят или не имеет IP, ВМ не смогут общаться друг с другом.

## **1.1. Рекомендованные параметры CPU / RAM / Disk**

### **Что делаем**  
Определяем минимальные и рекомендуемые параметры виртуальных машин для тестового кластера Kubernetes The Rough Way.

### **Почему так**  
Kubernetes — это не игрушка. Даже тестовый кластер требует ресурсов, иначе:

- kubelet будет падать в OOM,  
- containerd будет зависать,  
- etcd будет лагать и терять лидерство,  
- kube‑apiserver будет отвечать с задержками,  
- scheduler будет работать нестабильно.

Мы не хотим, чтобы человек думал, что у него “сломался Kubernetes”, когда у него просто 1 ГБ RAM.

### **Рекомендации (минимум / оптимум)**  
**admin‑wks:**  
- CPU: 1–2 vCPU  
- RAM: 2–4 GB  
- Disk: 20 GB  
- Почему: это просто управляющая машина, но на ней будут храниться бинарники, PKI, скрипты.

**cp‑01 (шаблон):**  
- CPU: 2 vCPU (минимум), лучше 4  
- RAM: 4 GB минимум  
- Disk: 40 GB  
- Почему: etcd + kube‑apiserver + controller‑manager + scheduler → всё это требует памяти.

**worker‑01 (шаблон):**  
- CPU: 2 vCPU  
- RAM: 4 GB  
- Disk: 40 GB  
- Почему: containerd + kubelet + Pod workload.

### **CKA‑ремарка**  
На экзамене CKA часто дают задачи, где кластер тормозит из‑за нехватки ресурсов. Понимание минимальных требований — часть troubleshooting.

## **1.2. Подключение к vmnet**

### **Что делаем**  
При создании каждой ВМ выбираем **Custom → vmnetXX** (или vmnet8 на Windows).

### **Почему так**  
- Все ноды должны быть в **одном L2‑сегменте**.  
- VRRP (VIP) работает только в одном широковещательном домене.  
- Calico BGP требует прямой L3‑связности.  
- kube‑proxy IPVS требует корректного ARP.

Если ноды окажутся в разных сетях — кластер не поднимется.

### **CKA‑ремарка**  
Понимание L2/L3 — важная часть troubleshooting на экзамене.

## **1.5. LVM‑разметка (минимальные размеры)**

### **Что делаем**  
Во время установки Ubuntu создаём LVM‑разметку согласно минимальным требованиям проекта.

### **Почему так**  
- etcd требует отдельного LV → иначе при нагрузке может “забить” корень.  
- containerd создаёт много файлов → нужен отдельный LV.  
- kubelet хранит Pod‑данные → отдельный LV.  
- `/var/log` должен быть отдельным → иначе journald может убить систему.

### **Минимальные размеры (для тестового кластера)** 

**Control Plane Node (cp‑01):**  
- `/` — 10 GB  
- `/var/log` — 4 GB  
- `/var/lib/etcd` — 8 GB  
- `/var/lib/kubelet` — 6 GB  
- `/var/lib/containerd` — 10 GB  
- свободное место — остальное

**Worker Node (worker‑01):**  
- `/` — 10 GB  
- `/var/log` — 4 GB  
- `/var/lib/kubelet` — 8 GB  
- `/var/lib/containerd` — 12 GB  
- свободное место — остальное

### **Почему такие размеры**  
- Это тестовый кластер → не нужно 20–40 GB на etcd.  
- Но слишком маленькие LV приведут к DiskPressure.  
- Мы оставляем свободное место, чтобы **расширять LV через LVM** при необходимости.

### **CKA‑ремарка**  
Умение работать с LVM (resize, extend) — часть экзамена.

## **1.6. Настройка файла /etc/hosts**

### **Что делаем**  
На каждой из трёх базовых ВМ (admin‑wks, cp‑01, worker‑01) приводим файл `/etc/hosts` к единому, предсказуемому виду, включающему:

- короткие имена нод,  
- полные доменные имена (FQDN),  
- VIP для kube‑apiserver,  
- все control plane ноды,  
- все worker‑ноды,  
- admin‑wks.

**Почему так**

#### **1. Kubernetes компоненты используют DNS‑имена**
- kubelet регистрирует ноду по FQDN;  
- kube‑apiserver использует SAN‑записи в сертификатах;  
- etcd требует FQDN в `--initial-cluster`;  
- CoreDNS должен резолвить ноды.

#### **2. FQDN обязателен для PKI**
В сертификатах kube‑apiserver должны быть SAN:

- `cp-01.k8s.local`  
- `cp-02.k8s.local`  
- `cp-03.k8s.local`  
- `k8s-vip.k8s.local`  

Без этого kubectl будет ругаться на TLS.

#### **3. /etc/hosts — источник правды при отсутствии DNS**
В лабораторной среде нет корпоративного DNS, поэтому `/etc/hosts` — единственный стабильный механизм резолвинга.

#### **4. Логи и диагностика**
В логах kubelet и apiserver FQDN читается лучше, чем bare hostname.

---

### **Содержимое `/etc/hosts`**

Это **единый эталон**, который должен быть на всех нодах.

```bash
127.0.0.1   localhost
127.0.1.1   <hostname>.k8s.local <hostname>

# Kubernetes The Rough Way cluster
192.168.88.10   k8s-vip.k8s.local k8s-vip k8s-api
192.168.88.11   cp-01.k8s.local cp-01
192.168.88.12   cp-02.k8s.local cp-02
192.168.88.13   cp-03.k8s.local cp-03
192.168.88.21   worker-01.k8s.local worker-01
192.168.88.22   worker-02.k8s.local worker-02
192.168.88.9    admin-wks.k8s.local admin-wks
```

### **Важно для клонов (cp‑02, cp‑03, worker‑02)**

После клонирования:

#### **1. Строка `127.0.1.1` должна быть изменена**
Например, на cp‑02:

```
127.0.1.1 cp-02.k8s.local cp-02
```

#### **2. IP‑адреса в нижнем блоке НЕ меняются**
Они одинаковые на всех нодах — это таблица резолвинга всего кластера.

#### **3. Hostname должен быть уникальным**
Иначе kubelet зарегистрирует ноду неправильно.

---

### **CKA‑ремарка**
На экзамене часто встречаются задачи, где:

- kubelet не стартует из‑за неправильного hostname;  
- kube‑apiserver ругается на SAN;  
- etcd не формирует кластер из‑за отсутствия FQDN;  
- CoreDNS не резолвит ноды.

Умение быстро проверить и поправить `/etc/hosts` — критически важный навык.

## **1.7. Проверка базовой доступности**

### **Что делаем**  
После установки ОС проверяем, что каждая машина:

- загружается корректно;  
- имеет рабочий статический IP;  
- может пинговать все остальные ноды по **IP**, **hostname**, **FQDN**;  
- имеет доступ в интернет;  
- принимает SSH‑подключения.

Эта проверка выполняется **на каждой из трёх базовых ВМ**:

- `admin-wks`  
- `cp-01`  
- `worker-01`

---

### **Проверка 1 — Пинг по IP**

Выполнить на **каждой** машине:

```bash
{
ping -c 1 192.168.88.9   # admin-wks
ping -c 1 192.168.88.11  # cp-01
ping -c 1 192.168.88.21  # worker-01
}
```

**Ожидаемый результат:** все пинги успешны.

---

### **Проверка 2 — Пинг по hostname**

```bash
{
ping -c 1 admin-wks
ping -c 1 cp-01
ping -c 1 worker-01
}
```

**Ожидаемый результат:** все имена резолвятся через `/etc/hosts`.

---

### **Проверка 3 — Пинг по FQDN (DNS‑name)**

```basg
{
ping -c 1 admin-wks.k8s.local
ping -c 1 cp-01.k8s.local
ping -c 1 worker-01.k8s.local

}
```

**Ожидаемый результат:** все FQDN резолвятся через `/etc/hosts`.

---

### **Проверка 4 — Проверка маршрутизации и ARP**

```bash
{
ip route
arp -n
}
```

**Ожидаемый результат:**

- маршрут по умолчанию указывает на `192.168.88.2`;  
- ARP‑таблица содержит соседей.

---

### **Проверка 5 — Проверка доступа в интернет**

```bash
{
ping -c 1 8.8.8.8
ping -c 1 google.com
}
```

**Ожидаемый результат:** интернет доступен.

---

### **Проверка 6 — Проверка SSH**

Вы должны иметь возможность подключиться к каждой виртуальной машине по ssh как с хостовой машины, так и с любой только что созданной нами виртуальной машине.

**Ожидаемый результат:** SSH работает, fingerprint принимается.
