# **8. Дисковая архитектура Kubernetes (LVM)**

## **8.1. Зачем вообще думать о дисках в Kubernetes**

В “учебных” кластерах всё ставят в один раздел `/`.  
В production это гарантированная смерть:

- **containerd** забивает диск образами и слоями;  
- **kubelet** забивает диск логами Pod’ов и временными файлами;  
- **/var/log** разрастается из‑за journald и системных сервисов;  
- **etcd** перестаёт писать WAL, теряет кворум, кластер падает;  
- нода уходит в `NotReady` из‑за `DiskPressure`.

В self‑hosted production‑ready Kubernetes **дисковая архитектура — это не косметика, а часть отказоустойчивости**.

Поэтому мы:

- используем **LVM**;  
- **жёстко разделяем** критичные директории;  
- **отдельно выделяем** etcd, kubelet, containerd;  
- оставляем **свободное место в VG** для расширения.

---

## **8.2. Почему LVM — базовый стандарт для self‑hosted Kubernetes**

LVM даёт:

- **гибкость** — `lvextend` без переустановки;  
- **изоляцию** — один компонент не убивает другой;  
- **контроль** — можно задать лимиты по размеру;  
- **возможность вынести LV на другой диск/хранилище**;  
- **снапшоты** — особенно важно для etcd.

Это соответствует:

- практикам Production Kubernetes, Kubernetes Best Practices;  
- рекомендациям SIG Node / SIG Etcd;  
- подходам OpenShift, GKE On‑Prem, RKE2, Tanzu.

---

## **8.3. Архитектура дисков Control Plane (LVM)**

Control Plane — это:

- **etcd** (единственный источник правды);  
- **kube-apiserver**;  
- **kube-controller-manager**;  
- **kube-scheduler**;  
- **kubelet**;  
- **containerd**;  
- системные логи.

Здесь **ошибка в дисковой архитектуре = риск потери кластера**.

### **8.3.1. Базовая схема разбиения диска (пример: 40 GB)**

> Размеры можно адаптировать под реальные диски, важна **структура и изоляция**, а не конкретные числа.

- `/dev/sda1` — **/boot**, 1 GB, без LVM, `ext4`.  
- `/dev/sda2` — **LVM PV**, Volume Group `vg_system`, остальное пространство.

### **8.3.2. Logical Volumes для Control Plane**

Во `vg_system` создаём:

- **LV `/` — 8–10 GB**  
  ОС, systemd, базовые утилиты.  
  Здесь **не должно** жить ничего тяжёлого.

- **LV `/var/log` — 4–6 GB**  
  Системные логи, journald, логи сервисов.  
  Если `/var/log` забьётся — ОС продолжит жить, etcd и containerd не пострадают.

- **LV `/var/lib/etcd` — 4–8 GB**  
  **Священная земля.**  
  Здесь живёт база etcd и WAL.  
  Требования:

  - минимальная латентность fsync;  
  - отсутствие конкуренции с containerd/kubelet;  
  - предсказуемый I/O.

- **LV `/var/lib/kubelet` — 4–6 GB**  
  kubelet хранит:

  - pod‑spec’ы;  
  - тома;  
  - часть временных данных.

  Если kubelet забьёт диск — нода не должна убить `/` или `/var/lib/etcd`.

- **LV `/var/lib/containerd` — 4–8 GB**  
  containerd хранит:

  - слои образов;  
  - unpacked rootfs;  
  - overlayfs.

  Это один из самых “прожорливых” по диску компонентов.

- **Свободное место во VG — 2–4 GB**  
  Резерв под `lvextend`:

  - если etcd начнёт расти;  
  - если containerd начнёт съедать больше;  
  - если `/var/log` начнёт забиваться.

### **8.3.3. Почему именно так**

- **etcd** — отдельный LV → защита от I/O‑шторма containerd/kubelet.  
- **kubelet** — отдельный LV → Pod’ы не убивают `/`.  
- **containerd** — отдельный LV → образы не убивают `/` и `/var/log`.  
- **/var/log** — отдельный LV → journald не убивает ОС и не мешает etcd.

## **8.4. Архитектура дисков Worker Nodes (LVM)**

Worker‑ноды не хостят etcd, но:

- выполняют Pod’ы;  
- хранят образы;  
- пишут логи;  
- могут быть под нагрузкой CI/CD.

### **8.4.1. Базовая схема разбиения диска (пример: 40–60 GB)**

- `/dev/sda1` — **/boot**, 1 GB.  
- `/dev/sda2` — **LVM PV**, Volume Group `vg_worker`, остальное пространство.

### **8.4.2. Logical Volumes для Worker Nodes**

Во `vg_worker` создаём:

- **LV `/` — 8–10 GB**  
  ОС и базовые утилиты.

- **LV `/var/log` — 4–6 GB**  
  Системные логи.

- **LV `/var/lib/kubelet` — 6–10 GB**  
  kubelet‑данные, тома, временные файлы Pod’ов.

- **LV `/var/lib/containerd` — 15–30 GB**  
  Основной потребитель диска:

  - образы;  
  - слои;  
  - кеши.

- **Свободное место во VG — 2–4 GB**  
  Для расширения kubelet/containerd/логов.

### **8.4.3. Почему worker‑ноды без `/var/lib/etcd`**

- etcd живёт только на control plane.  
- Worker‑ноды не должны хранить критичные данные кластера.  
- Это упрощает бэкапы и восстановление.

## **8.5. Почему Kubernetes‑бинарники НЕ в `/opt`**

Ты отдельно просил это объяснить — делаем нормально.

### **8.5.1. FHS и роль Kubernetes**

По FHS:

- `/opt` — для **сторонних приложений**, обычно монолитных;  
- `/usr` и `/usr/local` — для **системных бинарников**.

Kubernetes в нашем проекте — это **часть системного уровня**, а не “ещё одно приложение”.

### **8.5.2. Где должны лежать бинарники Kubernetes**

Мы используем:

- **`/usr/local/bin`** — для:

  - `kube-apiserver`  
  - `kube-controller-manager`  
  - `kube-scheduler`  
  - `kubelet`  
  - `kube-proxy`  
  - `kubectl`  
  - `etcd`, `etcdctl`  

Причины:

- `/usr/local/bin` в `$PATH` по умолчанию;  
- systemd‑юниты ожидают системные бинарники именно там;  
- это не конфликтует с пакетными версиями (если они появятся);  
- это соответствует best practices self‑hosted Kubernetes.

### **8.5.3. Почему НЕ `/opt/kubernetes/bin`**

Проблемы:

- ломает FHS;  
- ломает ожидания systemd;  
- ломает привычный `$PATH`;  
- усложняет обновления;  
- создаёт путаницу между “системой” и “приложением”.

## **8.6. Почему `/var/lib/kubelet` и `/var/lib/containerd` — отдельные LV**

Теперь **без противоречий** с разметкой.

Это не просто “хотелка”, а рекомендация:

- Kubernetes SIG Node;  
- Red Hat OpenShift;  
- GKE On‑Prem;  
- RKE2;  
- Tanzu.

### Причины:

- **kubelet**:

  - хранит pod‑данные;  
  - создаёт тома;  
  - может разрастаться.

- **containerd**:

  - хранит образы;  
  - кеши;  
  - слои.

Если они живут на `/`:

- забивают корень → нода падает.  

Если они живут на том же LV, что и `/var/lib/etcd`:

- I/O‑шторма от Pod’ов → задержки fsync → проблемы с etcd.

**Изоляция через отдельные LV — единственный честный способ защитить кластер.**

---

## **8.7. Свободное место во VG — это не роскошь, а инструмент**

Мы **специально** оставляем 2–4 GB свободными в VG:

- чтобы можно было:

  - расширить `/var/lib/etcd`, если кластер растёт;  
  - расширить `/var/lib/containerd`, если CI/CD начал жечь диск;  
  - расширить `/var/log`, если логирование стало более подробным.

Это стандартная практика SRE:  
**не размечать диск “под ноль”, а оставлять манёвренность.**

---

## **8.8. Swap и его роль в дисковой архитектуре**

swap должен быть убит

### **8.8.1. Требование Kubernetes**

По умолчанию:

- kubelet **требует отключённый swap**,  
- иначе:

  - либо kubelet не стартует;  
  - либо работает в режиме, который не рекомендуется для production.

### **8.8.2. Почему swap off в production Kubernetes**

- предсказуемость latency;  
- отсутствие неожиданных swap‑штормов;  
- отсутствие влияния swap на etcd и control plane;  
- упрощение SLO по времени ответа.

### **8.8.3. Где это будет описано**

Подробная процедура:

- `swapoff -a`;  
- удаление swap из `/etc/fstab`;  
- проверка через `free -h` и `swapon --show`;

будет зафиксирована в **документе** — про подготовку ОС.  
Здесь мы только фиксируем **архитектурное решение**:  
**в нашем кластере swap всегда отключён.**

## **8.9. Итоговая сводка по LVM**

### Control Plane (пример)

- `/boot` — 1G, отдельный раздел.  
- `VG vg_system`:

  - `/` — 8–10G  
  - `/var/log` — 4–6G  
  - `/var/lib/etcd` — 4–8G  
  - `/var/lib/kubelet` — 4–6G  
  - `/var/lib/containerd` — 4–8G  
  - free — 2–4G

### Worker

- `/boot` — 1G.  
- `VG vg_worker`:

  - `/` — 8–10G  
  - `/var/log` — 4–6G  
  - `/var/lib/kubelet` — 6–10G  
  - `/var/lib/containerd` — 15–30G  
  - free — 2–4G
