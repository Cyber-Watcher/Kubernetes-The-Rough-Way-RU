# **7. Сетевая архитектура Kubernetes**

Сетевая архитектура — это скелет кластера.  
От неё зависят:

- связность Pod → Pod;  
- работа Service;  
- доступ к API;  
- работа Calico и kube-proxy;  
- корректность PKI и SAN;  
- поведение ingress.

В Kubernetes The Rough Way мы используем:

- **Node CIDR:** реальная сеть нод `192.168.88.0/24`;  
- **Pod CIDR:** `10.244.0.0/16`, разбитый по нодам;  
- **Service CIDR:** `10.96.0.0/12`;  
- **Cluster DNS IP:** `10.96.0.10`;  
- **Calico:** BGP, без overlay;  
- **kube-proxy:** IPVS;  
- **API:** VIP `192.168.88.10` + HAProxy + Keepalived.

Документ описывает, как всё это связано.

---

## **Нюанс работы Calico BGP без Overlay и варианты оверлея**

В Kubernetes The Rough Way мы используем **Calico в режиме “Pure BGP (No-Encap)”**:

- нет VXLAN;  
- нет IP-in-IP (IPIP);  
- Pod → Pod трафик идёт как чистый L3;  
- каждая нода анонсирует свой PodCIDR через BGP.

Это работает **“из коробки” только потому, что весь кластер живёт в одном L2‑сегменте**:

```text
Node CIDR: 192.168.88.0/24

cp‑01:     192.168.88.11
cp‑02:     192.168.88.12
cp‑03:     192.168.88.13
worker‑01: 192.168.88.21
worker‑02: 192.168.88.22
```

Все ноды:

- находятся в одной подсети `192.168.88.0/24`;  
- видят друг друга напрямую;  
- внешняя сеть не знает о `10.244.x.x`, но это и не нужно — весь Pod‑трафик ходит только между этими нодами.

Это **критический инвариант текущей архитектуры**.

---

### Если появляются ноды в другой подсети

Допустим, мы добавляем новые worker‑ноды в другую подсеть:

```text
Новая подсеть: 192.168.99.0/24

worker‑03: 192.168.99.21
worker‑04: 192.168.99.22
```

Тогда:

- Pod’ы на `192.168.88.x` и `192.168.99.x` имеют IP из `10.244.x.x`;  
- без допнастроек физическая сеть и маршрутизатор **не знают путей до 10.244.x.x**;  
- Pod → Pod трафик между подсетями `88` и `99` начнёт теряться.

Чтобы это исправить, есть **три режима Calico**.

---

### 1. **Pure BGP (No-Encap)** — как у нас сейчас

- нет оверлея;  
- маршруты `10.244.x.x` распространяются через BGP;  
- работает идеально, если:
  - все ноды в одном L2‑сегменте, **или**
  - есть BGP‑пиринг с L3‑маршрутизатором, который принимает PodCIDR и раздаёт их дальше.

Для случая с `192.168.88.0/24` + `192.168.99.0/24` это значит:

- Calico на нодах `88` и `99` поднимает BGP‑сессию с маршрутизатором;  
- маршрутизатор знает, где находятся `10.244.21.0/24`, `10.244.22.0/24`, `10.244.31.0/24`, `10.244.32.0/24`;  
- Pod → Pod трафик между подсетями ходит прозрачно.

---

### 2. **BGP + IP-in-IP (IPIP)** — BGP с оверлеем

Этот режим нужен, когда:

- ноды в разных L3‑подсетях;  
- промежуточные роутеры **не знают** о `10.244.x.x`;  
- но BGP между нодами всё равно доступен.

Схема:

- **Control Plane (BGP):** ноды по BGP договариваются: “PodCIDR X за мной, PodCIDR Y за тобой”;  
- **Data Plane (IPIP):** пакет пода упаковывается в внешний IP‑пакет с адресами **IP нод**.

На проводе это выглядит так:

```text
[ L2 ] [ IP Ноды‑A → IP Ноды‑B ] [ IP Пода‑A → IP Пода‑B ] [ Payload ]
```

Для роутера это обычный трафик между серверами `192.168.88.x` и `192.168.99.x`.  
Calico на принимающей ноде снимает внешний “конверт” и отдаёт внутренний пакет поду.

Итог:  
BGP знает, куда слать туннель, IPIP делает оверлей, роутерам не нужно знать о `10.244.x.x`.

---

### 3. **VXLAN (без BGP)**

В Calico режим VXLAN:

- **не использует BGP вообще**;  
- маршруты и туннели управляются через Kubernetes API/etcd;  
- внешняя сеть видит только Node IP;  
- Pod → Pod трафик идёт внутри VXLAN‑туннелей.

Разделение в Calico жёсткое:

1. **VXLAN** — всегда оверлей, BGP отключен.  
2. **IPIP** — оверлей, маршруты распространяются через BGP.  
3. **Pure BGP (No-Encap)** — без оверлея, маршруты через BGP.

---

### Итог для нашей архитектуры

Сейчас:

- все ноды в `192.168.88.0/24`;  
- мы используем **Pure BGP (No-Encap)** — без оверлея;  
- это работает идеально в одном L2‑сегменте.

Если в будущем появятся worker‑ноды в других подсетях (например, `192.168.99.0/24`), чтобы сохранить Pod‑связность между ними, придётся:

- либо настроить **BGP‑пиринг с L3‑маршрутизатором** (оставив Pure BGP),  
- либо включить **IPIP (BGP + оверлей)**,  
- либо перейти на **VXLAN**, если BGP на маршрутизаторе невозможен.

---

## **7.1. Pod CIDR**

Pod CIDR — это диапазон IP‑адресов, из которого Calico выдаёт IP Pod’ам.

В нашем кластере:

```text
Pod CIDR: 10.244.0.0/16
```

Мы делим его на подсети /24 **по нодам**:

- `cp‑01`: `10.244.11.0/24`  
- `cp‑02`: `10.244.12.0/24`  
- `cp‑03`: `10.244.13.0/24`  
- `worker‑01`: `10.244.21.0/24`  
- `worker‑02`: `10.244.22.0/24`  

Каждая нода:

- получает свой PodCIDR;  
- анонсирует его через BGP;  
- маршрутизирует Pod‑трафик напрямую, без encapsulation.

CP‑ноды тоже имеют PodCIDR, потому что:

- на них работает kubelet;  
- на них работает kube-proxy;  
- на них работает Calico DaemonSet;  
- они участвуют в сетевой плоскости, даже если не запускают workload.

---

## **7.2. Service CIDR**

Service CIDR — это виртуальная сеть для Service IP.

В нашем кластере:

```text
Service CIDR: 10.96.0.0/12
```

Особенности:

- Service‑IP **не существуют в физической сети**;  
- Service‑IP **не маршрутизируются** как обычные IP;  
- реализация Service происходит через **kube-proxy IPVS** на каждой ноде.

Service‑IP — это чистая абстракция Kubernetes, реализованная на нодах.

---

## **7.3. Cluster DNS IP**

DNS‑сервис CoreDNS получает фиксированный IP из Service CIDR:

```text
Cluster DNS IP: 10.96.0.10
```

Этот IP:

- прописывается в конфигурации kubelet;  
- используется Pod’ами как DNS‑сервер;  
- резолвит `*.svc.cluster.local`.

Инвариант:

- `10.96.0.10` всегда указывает на CoreDNS Service;  
- CoreDNS всегда доступен внутри кластера по этому IP.

---

## **7.4. Node CIDR**

Node CIDR — это сеть **самих нод**, их реальных IP в физической сети.

В нашем кластере:

```text
Node CIDR: 192.168.88.0/24
gateway:   192.168.88.2
VIP:       192.168.88.10
```

Размещение IP:

- `cp‑01`: `192.168.88.11`  
- `cp‑02`: `192.168.88.12`  
- `cp‑03`: `192.168.88.13`  
- `worker‑01`: `192.168.88.21`  
- `worker‑02`: `192.168.88.22`  

### Инварианты Node CIDR

- Node IP — это **реальный IP**, доступный в физической сети.  
- kubelet, containerd, apiserver, etcd используют Node IP как точку входа.  
- Node CIDR должен быть **маршрутизируемым** внутри L2/L3 домена.  
- VIP `192.168.88.10` живёт в этом же CIDR.

---

## **7.5. Поток трафика Pod → Service → Pod**

Ключевой поток, который показывает взаимодействие Calico и kube-proxy.

```text
Pod A (10.244.21.15) на worker‑01
        │
        ▼
Service (10.96.0.20)
        │
        ▼
kube-proxy (IPVS) на worker‑01
        │
        ▼
DNAT → Pod B (10.244.22.33) на worker‑02
        │
        ▼
Calico BGP маршрутизация
```

Пошагово:

1. Pod A отправляет пакет на Service IP `10.96.0.20`.  
2. kube-proxy IPVS на `worker‑01` делает DNAT: `10.96.0.20 → 10.244.22.33`.  
3. Calico смотрит маршрут до `10.244.22.33` и отправляет пакет на `worker‑02`.  
4. Pod B получает пакет напрямую.

Особенности:

- нет VXLAN/IPIP;  
- нет encapsulation overhead;  
- чистый L3 между нодами;  
- Service‑IP существует только как IPVS‑правило на нодах.

---

## **7.6. Поток трафика kubelet → apiserver**

Все kubelet (и на CP, и на worker‑нодах) подключаются к API **только через VIP**.

```text
kubelet (192.168.88.21)
        │
        ▼
VIP 192.168.88.10 (k8s-vip / api.k8s.local)
        │
        ▼
HAProxy на одной из CP‑нод
        │
        ▼
kube-apiserver (192.168.88.11/12/13)
```

Инварианты:

- kubelet всегда использует VIP `192.168.88.10`;  
- SAN сертификата kube-apiserver включает VIP, IP всех CP‑нод и все DNS;  
- kubelet использует клиентский сертификат, подписанный Kubernetes CA;  
- HAProxy балансирует трафик между `cp‑01`, `cp‑02`, `cp‑03`.

---

## **7.7. Поток трафика etcd peer → peer**

etcd‑кластер общается напрямую между нодами, без VIP.

```text
etcd‑1 (192.168.88.11) ↔ etcd‑2 (192.168.88.12) ↔ etcd‑3 (192.168.88.13)
```

Особенности:

- используется **peer TLS**;  
- SAN etcd‑peer включает IP и DNS каждой ноды;  
- трафик идёт по Node CIDR `192.168.88.0/24`;  
- никакого VIP, никакого round‑robin.

Инварианты:

- etcd всегда знает конкретные IP peer‑нод;  
- DNS может использоваться, но IP в SAN обязателен;  
- стабильность Raft зависит от стабильности этого трафика.

---

## **7.8. Поток трафика ingress → service → pod**

Путь внешнего HTTP/HTTPS‑запроса до Pod’а.

```text
Client
  │
  ▼
Ingress (NGINX Ingress Controller)
  │
  ▼
Service (10.96.x.x)
  │
  ▼
kube-proxy (IPVS)
  │
  ▼
Pod (10.244.x.x)
```

Пошагово:

1. Клиент подключается к ingress‑узлу (NodePort/hostPort/LoadBalancer).  
2. NGINX Ingress Controller выбирает Service по правилам Ingress.  
3. kube-proxy IPVS делает DNAT Service → Pod.  
4. Calico маршрутизирует трафик к нужной ноде по PodCIDR.

Инварианты:

- Ingress никогда не ходит напрямую к Pod’ам;  
- всегда через Service;  
- Service всегда реализуется через kube-proxy;  
- маршрутизация Pod‑IP всегда через Calico.

---

## **7.9. Почему Calico ставится на Control Plane**

Calico — это DaemonSet, и он ставится на **все ноды**, включая CP.

Причины:

- kubelet работает на CP‑нодах → нужен CNI;  
- kube-proxy работает на CP‑нодах → нужен CNI;  
- системные DaemonSet’ы (node-exporter, metrics-server и т.п.) могут запускаться на CP;  
- PodCIDR должен быть у каждой ноды;  
- BGP должен видеть все ноды как участники Pod‑сети.

CP‑ноды:

- не запускают пользовательский workload (taints);  
- но являются полноценной частью сетевой плоскости.

---

## **7.10. Инварианты сетевой архитектуры**

- **Node CIDR:** `192.168.88.0/24`  
- **Gateway:** `192.168.88.2`  
- **VIP:** `192.168.88.10` (`k8s-vip`, `api.k8s.local`)  
- **CP‑ноды:** `.11`, `.12`, `.13`  
- **Worker‑ноды:** `.21`, `.22`  

- **Pod CIDR:** `10.244.0.0/16`  
  - `cp‑01`: `10.244.11.0/24`  
  - `cp‑02`: `10.244.12.0/24`  
  - `cp‑03`: `10.244.13.0/24`  
  - `worker‑01`: `10.244.21.0/24`  
  - `worker‑02`: `10.244.22.0/24`  

- **Service CIDR:** `10.96.0.0/12`  
- **Cluster DNS IP:** `10.96.0.10`  

- Calico: BGP, без overlay;  
- kube-proxy: IPVS;  
- все компоненты ходят к API через VIP;  
- SAN kube-apiserver включает все IP и DNS;  
- etcd использует отдельный CA и peer TLS по Node IP.

---

## **7.11. Итоговая схема сетевой архитектуры**

```text
                 Физическая сеть (Node CIDR)
              192.168.88.0/24 (gateway 192.168.88.2)

        VIP 192.168.88.10 (k8s-vip / api.k8s.local)
                     │
     ┌───────────────┼─────────────────────────────┐
     ▼               ▼                             ▼
  cp‑01           cp‑02                         cp‑03
  88.11           88.12                         88.13
  PodCIDR 11/24   PodCIDR 12/24                 PodCIDR 13/24
     │               │                             │
     └───────────────┼───────────────┬─────────────┘
                     │               │
                 BGP (Calico)   BGP (Calico)
                     │               │
     ┌───────────────┴───────────────┴─────────────┐
     ▼                                             ▼
 worker‑01                                      worker‑02
 192.168.88.21                                192.168.88.22
 PodCIDR 10.244.21.0/24                PodCIDR 10.244.22.0/24
```

---

## **7.12. Итог**

Сетевая архитектура Kubernetes The Rough Way:

- чётко разделяет Node CIDR, Pod CIDR и Service CIDR;  
- использует Calico BGP без overlay для Pod‑сети;  
- использует kube-proxy IPVS для Service‑сети;  
- использует VIP для HA‑доступа к API;  
- согласована с PKI (SAN, CA, kubeconfig);  
- согласована с Control Plane, Worker Nodes;  
- отражает реальную продакшен‑архитектуру bare‑metal кластера.
