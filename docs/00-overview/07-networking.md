# **7. Сетевая архитектура Kubernetes**

## **7.1. Pod CIDR**

Pod CIDR — это диапазон IP‑адресов, из которого выделяются адреса для Pod’ов на каждой ноде.  
В нашем проекте используется:

- **Pod CIDR: `10.200.0.0/16`**  
- Каждой ноде выделяется собственный /24:  
  - node1 → `10.200.1.0/24`  
  - node2 → `10.200.2.0/24`  
  - node3 → `10.200.3.0/24`  
  - и т.д.

### Инварианты Pod CIDR

- Pod IP **уникален в пределах всего кластера**.  
- Pod IP **не маршрутизируется за пределы кластера**.  
- Pod IP **не должен пересекаться** с сетью хоста, Service CIDR, VIP, Node CIDR.

### Почему /16

- позволяет выделить до 256 нод по /24  
- каждая нода получает до 256 Pod’ов  
- соответствует масштабу проекта

---

## **7.2. Service CIDR**

Service CIDR — диапазон виртуальных IP‑адресов для сервисов Kubernetes.

В проекте используется:

- **Service CIDR: `10.32.0.0/24`**  
- **Cluster DNS IP: `10.32.0.10`**

### Инварианты Service CIDR

- Service IP **не существует в реальной сети** — это виртуальный IP.  
- kube-proxy или eBPF‑плоскость (в нашем случае Calico) создаёт правила DNAT/SNAT.  
- Service IP **не должен пересекаться** с Pod CIDR, Node CIDR, VIP.

---

## **7.3. Cluster DNS IP**

Cluster DNS — это сервис CoreDNS.

Мы используем:

- **Cluster DNS IP: `10.32.0.10`**  
- Это первый доступный IP в Service CIDR (после зарезервированных).

### Почему именно .10

- исторически принято в Kubernetes  
- легко запомнить  
- не конфликтует с системными сервисами  
- используется kubelet по умолчанию

---

## **7.4. Node CIDR**

Node CIDR — это IP‑адреса самих нод.

В проекте используется:

- **Node CIDR: `192.168.88.0/24`**  
- gateway: `192.168.88.2`  
- VIP: `192.168.88.10`  
- control-plane: `.11`, `.12`, `.13`  
- workers: `.21`, `.22`, `.23`

### Инварианты Node CIDR

- Node IP — это **реальный IP**, доступный в физической сети.  
- kubelet, containerd, apiserver, etcd — используют Node IP как точку входа.  
- Node CIDR должен быть **маршрутизируемым** внутри L2/L3 домена.

---

## **7.5. Поток трафика Pod → Service → Pod**

Это ключевой путь в Kubernetes.

**1. Pod отправляет запрос на Service IP**  
Например:

```
10.200.1.15 → 10.32.0.42:80
```

**2. kube-proxy/eBPF перехватывает пакет**  
- определяет, что 10.32.0.42 — виртуальный IP  
- выбирает backend Pod по алгоритму (round-robin, random, session affinity)

3. Выполняется DNAT  
```
10.200.1.15 → 10.200.3.25:8080
```

**4. Calico маршрутизирует пакет между нодами**  
- через BGP  
- без оверлея  
- без encapsulation  
- L3‑маршрутизация между Pod CIDR’ами

**5. Ответ возвращается напрямую (DSR)**  
Если включён BGP, ответ идёт **напрямую**, минуя сервис.

---

## **7.6. Поток трафика kubelet → apiserver**

kubelet общается с apiserver по HTTPS.

**Поток:**

```
kubelet (node) → VIP 192.168.88.10:6443 → apiserver (control-plane)
```

### Инварианты:

- kubelet использует **kubeconfig с клиентским сертификатом**  
- VIP обеспечивает HA  
- apiserver отвечает только по HTTPS  
- kubelet не общается с etcd напрямую

---

## **7.7. Поток трафика etcd peer → peer**

etcd использует протокол Raft.

**Поток:**

```
etcd1 → etcd2 → etcd3 → etcd1
```

**Порты:**

- 2380 — peer communication  
- 2379 — client communication (apiserver)

### Инварианты:

- каждый etcd имеет свой SAN в сертификате  
- трафик peer‑to‑peer всегда шифрован  
- quorum = 2 из 3  
- потеря одной ноды не нарушает работу

---

## **7.8. Поток трафика ingress → service → pod**

Когда внешний клиент обращается к приложению:

**1. Клиент → ingress‑контроллер** 
Например:

```
client → 192.168.88.10:80 → ingress
```

**2. Ingress выбирает backend service**
Например:

```
/api → service api-svc (10.32.0.42)
```

**3. Service перенаправляет на Pod**  
```
10.32.0.42 → 10.200.3.25:8080
```

**4. Pod отвечает напрямую**  
Calico маршрутизирует ответ обратно.
