# **5. Worker Nodes — детальный обзор**

Этот документ описывает компоненты worker‑нод, их роли, взаимодействие с control plane, сетевую модель, жизненный цикл Pod’ов и то, как worker‑ноды вписываются в архитектуру Kubernetes The Rough Way.

Worker‑ноды — это **workload plane (рабочая плоскость)**.  
Именно здесь запускаются Pod’ы приложений, системные DaemonSet’ы, сетевые агенты и сервисы, которые обслуживают реальную нагрузку.

В нашем кластере есть две worker‑ноды:

- `worker‑01`  
- `worker‑02`

Они идентичны по роли, сетевому стеку и LVM‑разметке.

---

## **5.1. Роль worker‑нод в Kubernetes**

Worker‑ноды выполняют:

- запуск Pod’ов и контейнеров;
- выполнение workload’ов (приложений);  
- управление жизненным циклом workload’ов;  
- сетевую маршрутизацию;  
- реализацию Service networking через kube-proxy;  
- применение NetworkPolicy через Calico;  
- взаимодействие с control plane через kube-apiserver.

Worker‑ноды:

- **не** хранят состояние кластера (это делает etcd на CP‑нодах);  
- **не** принимают решения о планировании (это делает kube-scheduler);  
- **не** выполняют встроенные контроллеры (это делает kube-controller-manager).

Они — исполнительный слой, который реализует решения, принятые control plane.

---

## **5.2. Компоненты worker‑ноды**

Каждая worker‑нода содержит четыре ключевых компонента:

- **kubelet**  
- **containerd**  
- **kube-proxy**  
- **CNI (Calico)**

Вместе они обеспечивают полный цикл работы Pod’ов:

- создание sandbox’а;  
- запуск контейнеров;  
- подключение к сети;  
- маршрутизацию трафика;  
- обновление статуса в API.

---

## **5.3. kubelet — агент ноды**

kubelet — главный агент worker‑ноды.

**Функции:**

- получает PodSpec от kube-apiserver;  
- создаёт Pod sandbox и network namespace;  
- вызывает CNI (Calico) для настройки сети Pod’а;
- взаимодействует с CNI;  
- запускает контейнеры через containerd (CRI);  
- следит за состоянием Pod’ов и контейнеров;  
- обновляет статус Pod’ов в kube-apiserver;  
- управляет readiness/liveness probes;  
- монтирует тома и временные директории.

**Особенности в нашем кластере:**

- работает на всех worker‑нодах и CP‑нодах (но workload’ы запускаются только на worker‑нодах);  
- подключается к API только через VIP `192.168.88.10`;  
- использует TLS‑сертификаты, выданные Kubernetes CA;  
- взаимодействует с containerd через `unix:///run/containerd/containerd.sock`;  
- хранит данные в `/var/lib/kubelet` (отдельный LV).

---

## **5.4. containerd — контейнерный runtime**

containerd — это контейнерный runtime, который непосредственно запускает контейнеры.

**Функции:**

- хранит образы контейнеров;  
- unpack rootfs и управляет overlayfs;  
- создаёт и удаляет контейнеры по запросу kubelet;
- управляет их жизненным циклом;  
- использует `runc` (или другой low-level runtime) для запуска контейнеров;  
- предоставляет CRI‑интерфейс для kubelet.

**Особенности в нашем кластере:**

- используется как **единственный runtime** (Docker не используется);  
- работает как systemd‑сервис;  
- взаимодействует с kubelet по gRPC;  
- хранит данные в `/var/lib/containerd` (отдельный LV).

---

## **5.5. kube-proxy — реализация Service networking**

kube-proxy — сетевой компонент, который реализует абстракцию Kubernetes Service.

**Функции:**

- создаёт правила DNAT для Service IP → Pod IP;  
- реализует типы Service: ClusterIP, NodePort, LoadBalancer;  
- синхронизирует список backend’ов на основе Endpoints;  
- обновляет правила при изменении Pod’ов и Endpoints.

**Особенности в нашем кластере:**

- работает в режиме **IPVS** (а не iptables);  
- получает конфигурацию от kube-apiserver;  
- не занимается маршрутизацией Pod‑IP между нодами (это делает Calico);
- обновляет правила при изменении Endpoints;  
- работает на всех нодах, включая control plane.

---

## **5.6. Calico (CNI) — Pod‑сеть и BGP‑маршрутизация**

Calico — это CNI‑плагин и сетевой стек, который обеспечивает Pod‑сеть и маршрутизацию.

**Функции:**

- создаёт veth‑пары и network namespace для Pod’ов;  
- назначает Pod‑IP (IPAM);  
- маршрутизирует Pod‑IP между нодами;  
- применяет NetworkPolicy;  
- анонсирует Pod CIDR ноды через BGP.

Мы используем **Calico BGP mode (без overlay)**. Это значит:

- каждый узел — BGP speaker;  
- каждый узел анонсирует свой PodCIDR;  
- маршруты Pod → Pod — чистый L3, без VXLAN/IPIP;  
- нет encapsulation overhead;  
- проще интеграция с физической сетью.

**Особенности в нашем кластере:**

- общий Pod CIDR: `10.244.0.0/16`;  
- `worker‑01`: `10.244.1.0/24`;  
- `worker‑02`: `10.244.2.0/24`;  
- маршруты анонсируются через BGP на уровне нод;  
- Calico установлен на всех нодах (CP + worker), но workload’ы запускаются только на worker‑нодах.

---

## **5.7. LVM‑разметка worker‑нод**

Worker‑ноды используют LVM для изоляции критичных директорий и предотвращения DiskPressure.

**Рекомендуемая разметка:**

- `/boot` — отдельный раздел (без LVM);  
- Volume Group, например `vg_worker`:
  - `/` — 8–10 GB (только ОС и базовые утилиты);  
  - `/var/log` — 4–6 GB (логи системы и сервисов);  
  - `/var/lib/kubelet` — 6–10 GB (данные Pod’ов, тома, временные директории);  
  - `/var/lib/containerd` — 15–30 GB (образы и слои контейнеров);  
  - свободное место — 2–4 GB под `lvextend` при необходимости.

**Причины такой схемы:**

- containerd может быстро разрастаться из‑за образов и слоёв;  
- kubelet создаёт временные тома и директории, которые могут забить диск;  
- journald и логи приложений могут забить `/var/log`;  
- изоляция через LVM предотвращает падение ноды из‑за переполнения `/`;  
- свободное место позволяет гибко расширять нужный LV без простоя.

---

## **5.8. Жизненный цикл Pod’а на worker‑ноде**

Жизненный цикл Pod’а на worker‑ноде выглядит так:

1. **Scheduler выбирает ноду.**  
   `kube-scheduler` анализирует ресурсы, taints/tolerations, affinity и выбирает `worker‑01` или `worker‑02`.

2. **kubelet получает PodSpec.**  
   kubelet на выбранной ноде получает описание Pod’а от kube-apiserver.

3. **Создание Pod sandbox.**  
   kubelet создаёт network namespace и базовый sandbox для Pod’а.

4. **CNI (Calico) настраивает сеть.**  
   kubelet вызывает CNI ADD, Calico создаёт veth‑пару, назначает Pod‑IP и подключает Pod к Pod‑сети.

5. **containerd запускает контейнеры.**  
   kubelet через CRI просит containerd запустить контейнеры Pod’а.

6. **kubelet обновляет статус Pod’а.**  
   kubelet сообщает kube-apiserver, что Pod запущен, и обновляет `status`.

7. **kube-proxy обновляет правила Service.**  
   kube-proxy получает обновлённые Endpoints и настраивает IPVS для Service → Pod.

8. **Calico обновляет маршруты.**  
   Calico при необходимости обновляет BGP‑маршруты, чтобы другие ноды знали, где находится Pod.

---

## **5.9. Сетевые взаимодействия worker‑ноды**

Worker‑нода взаимодействует с несколькими сущностями:

**С kube-apiserver:**

- kubelet получает PodSpec и отправляет статус;
- обновление статуса;  
- kube-proxy получает список Service и Endpoints;  
- Calico получает информацию о Pod’ах, нодах, IPPool, NetworkPolicy.

**С etcd (косвенно):**

- worker‑нода **никогда** не общается с etcd напрямую;  
- все операции идут через kube-apiserver.

**С другими worker‑нодами:**

- через BGP‑маршруты Calico;  
- Pod → Pod трафик идёт по L3‑маршрутам без оверлея.

**С control plane:**

- kubelet → kube-apiserver (через VIP);  
- kube-proxy → kube-apiserver;  
- Calico → kube-apiserver.

---

## **5.10. Почему worker‑ноды не являются частью control plane**

Worker‑ноды:

- не принимают архитектурных решений;  
- не хранят состояние кластера;  
- не выполняют встроенные контроллеры;  
- не управляют API.

Они только:

- запускают workload’ы;  
- обеспечивают контейнеры (containerd);  
- обеспечивают сеть (Calico + kube-proxy);  
- выполняют решения, принятые control plane.

Control Plane — это мозг.  
Worker‑ноды — это мускулы.

---

## **5.11. Итоговая схема worker‑ноды**

```text
                 Control Plane
┌──────────────────────────────────────────────┐
│            kube-apiserver (VIP)              │
└──────────────────────────────────────────────┘
                  ▲           ▲
                  │           │
          kubelet │           │ kube-proxy
                  │           │
                  │           │
┌─────────────────┴───────────┴──────────────┐
│                 Worker Node                │
│                                            │
│  ┌──────────────┐                          │
│  │   kubelet    │──────────────────────────┘
│  └──────────────┘   (talks to apiserver)   │
│          │                                 │
│          ▼                                 │
│  ┌──────────────┐                          │
│  │  containerd  │                          │
│  └──────────────┘                          │
│                                            │
│  ┌──────────────┐                          │
│  │  kube-proxy  │──────────────────────────┘
│  └──────────────┘   (watches Services/EP)  │
│                                            │
│  ┌──────────────┐                          │
│  │   Calico     │ ←→ BGP → другие ноды     │
│  └──────────────┘                          │
└────────────────────────────────────────────┘
```
